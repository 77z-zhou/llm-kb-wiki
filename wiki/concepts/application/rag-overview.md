# RAG (检索增强生成) 概述

## 核心原理

RAG 采用**"先检索，后生成"**模式，将信息检索与文本生成结合来增强 LLM 能力。

### 工作流程

1. **Retrieve（检索）**：用户问题 → 向量数据库搜索 → 找出最相关的文档块
2. **Augment（增强）**：检索结果 + 原始问题 → 拼接成增强 Prompt
3. **Generate（生成）**：增强 Prompt → LLM → 更准确的回答

## 解决的三大核心问题

| 问题 | RAG 如何解决 |
|:-----|:-------------|
| 知识过时 | 连接可随时更新的外部知识库 |
| 幻觉 | 提供事实依据"锚定"回答 |
| 缺乏领域知识 | 轻松连接私有数据 |

## RAG vs 微调

| 维度 | RAG | 微调 |
|:-----|:----|:-----|
| 知识更新 | 只需更新数据库 | 需重新训练 |
| 可追溯性 | 可展示来源文档 | 黑盒 |
| 幻觉控制 | 有据可循 | 仍可能幻觉 |
| 成本 | 低 | 高 |
| 适用场景 | 注入事实性知识 | 改变行为/风格 |

## RAG vs 搜索系统

| 特征 | 搜索系统 | RAG 系统 |
|:-----|:---------|:---------|
| 任务 | 找文档 | 给答案 |
| 输出 | 文档列表 | 自然语言答案 |
| 用户角色 | 主动阅读总结 | 被动获得成品 |
| 核心组件 | 索引器+排序器 | 索引器+排序器**+生成器(LLM)** |

## 补充见解

- **RAG 不是万能药**：对于需要深度推理、风格迁移、代码能力等场景，微调仍然是必要的
- **RAG + 微调 的组合是当前最佳实践**：微调提升 LLM 的基础能力，RAG 提供实时知识
- **"搜索增强生成"可能比"检索增强生成"更准确**：很多生产级 RAG 系统实际上用的是传统搜索+LLM
