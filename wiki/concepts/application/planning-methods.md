# Agent 规划方法

规划能力是 Agent 智能水平的核心指标，决定了能否将复杂目标分解为可执行步骤。

## 方法层次（从简单到复杂）

### 1. 基于提示的隐式规划

| 方法 | 特点 | 局限 |
|:-----|:-----|:-----|
| **CoT** | "Let's think step by step"，线性推理 | 无法回溯，一步错全盘皆输 |
| **ReAct** | 将 CoT 与行动结合，动态调整 | 比纯 CoT 鲁棒，但仍线性 |

### 2. 基于搜索的显式规划

| 方法 | 结构 | 核心操作 | 适用场景 |
|:-----|:-----|:---------|:---------|
| **Tree of Thoughts (ToT)** | 树 | 分支 + 评估 + 剪枝 + 回溯 | 需要深思熟虑的问题 |
| **Graph of Thoughts (GoT)** | 图 | 分支 + **合并** + **循环迭代** | 需要整合多信息流 |

**ToT 工作流**：
1. 从初始问题生成多个并行思考路径（树的分支）
2. 用 LLM 自身对每个叶子节点打分评估
3. 选择最有希望的节点继续扩展（BFS/DFS）
4. 允许回溯：剪掉低分路径

**GoT 对 ToT 的扩展**：
- 允许不同分支**合并**——将多个子问题的解汇集
- 允许**循环**——迭代优化已有解
- 更灵活但实现更复杂

### 3. 基于任务分解的规划

- LLM 充当"规划器"，将主任务分解为依赖图/步骤列表
- 另一个"执行器" LLM 逐一完成子任务
- 结构清晰但缺乏对动态变化的适应性

## 补充见解

- **实际工程中 ReAct 仍然是主流**：ToT/GoT 计算开销大（需多次 LLM 调用维护搜索树），目前更多在学术研究中
- **规划能力的瓶颈在 LLM 自身的推理能力**：如果 LLM 推理不够强，再好的规划框架也无法弥补
- **混合策略是趋势**：简单子任务用 CoT 直接处理，复杂子任务才启动搜索式规划
