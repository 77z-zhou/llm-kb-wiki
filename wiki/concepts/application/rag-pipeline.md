# RAG 完整流水线

## 阶段一：离线数据准备（索引流水线）

```
数据加载 → 文本切块 → 嵌入向量化 → 存入向量数据库
```

1. **数据加载 (Load)**：PDF/Word/网页/数据库 → 原始文档
2. **文本切块 (Chunk)**：长文档 → 语义完整的文本块
3. **嵌入 (Embed)**：文本块 → 高维向量（BERT/BGE/M3E）
4. **存储 (Store)**：向量+原文 → FAISS/ChromaDB/Pinecone

## 阶段二：在线查询（推理流水线）

```
用户提问 → 查询嵌入 → 向量检索 → [重排序] → 增强 Prompt → LLM 生成
```

1. **查询嵌入**：用相同嵌入模型将问题转为向量
2. **向量检索**：余弦相似度 → Top-K 文本块
3. **重排序 (可选)**：Cross-Encoder 精细打分 → Top-N
4. **增强生成**：文本块 + 问题 → LLM → 最终回答

## 文本切块策略

### 切块大小选择依据
- 嵌入模型的 Token 上限
- 数据类型（结构化按段落，非结构化按固定长度，代码按函数）
- 查询类型（具体问题→小块，宽泛问题→大块）

### 重叠长度
- 防止语义在边界被切断
- 经验法则：切块大小的 **10%-20%**

### 权衡

| | 大块 | 小块 |
|:--|:-----|:-----|
| 优点 | 上下文丰富 | 信息密度高，检索精确 |
| 缺点 | 噪声多，检索精度降 | 上下文不足，语义割裂 |

## 部署挑战

1. **数据管道复杂性**：异构数据源的分块策略难以泛化
2. **知识库实时更新**：增删改的 ETL 同步机制
3. **延迟与成本**：检索+生成两步天然比直接调用慢
4. **"无答案"处理**：检索结果不相关时应拒绝回答

## 补充见解

- **切块是 RAG 最需经验的步骤**：不同数据类型需要不同策略，没有万能方案
- **多尺度切块是趋势**：同时索引不同大小的块，应对不同粒度查询
- **"Lost in the Middle" 问题**：LLM 倾向忽略长上下文中间的信息，影响 RAG 效果
