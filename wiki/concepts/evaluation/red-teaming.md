# 红队测试与持续监控

## 红队测试 (Red Teaming)

对抗性测试方法：组织专门团队像"攻击者"一样主动寻找 LLM/Agent 的漏洞。

### 发现安全漏洞
- **绕过安全护栏**：设计越狱提示诱导生成有害内容
- **Prompt Injection**：恶意指令劫持 Agent 控制流（泄露信息/滥用工具）
- **资源滥用**：让 Agent 陷入无限循环或高消耗操作

### 发现偏见
- 暴露刻板印象（种族/性别/职业）
- 测试政治与社会偏见
- 揭示代表性不足问题

## 人工评估设计

### 评估准则 (Rubric)
1. **原子化维度**：准确性/完整性/简洁性/安全性（不用模糊的"好/坏"）
2. **量化评分**：1-5 分李克特量表，每级有清晰定义
3. **丰富示例**：每个分数等级提供正反面典型例子

### 评估流程
1. **标注者培训与校准会**
2. **盲评**：不告知来源模型
3. **多次独立评估**：2-3 名标注者，计算 IAA (Cohen's Kappa)
4. **成对比较**优于绝对评分
5. **仲裁机制**处理分歧

## 持续监控体系

```
采集日志+用户反馈 → 自动化监控 → 人工审核分析 → 反馈闭环+模型迭代
```

### 监控指标
- **输入**：问题长度、主题分布
- **输出**：回答长度、格式错误率、拒绝率
- **过程(Agent)**：平均步数、工具调用频率/失败率

### 自动化评估
- 定期抽样 + LLM-as-a-Judge 打分
- 与"黄金评估集"对比稳定性

## 补充见解

- **红队测试是"免疫系统压力测试"**：通过模拟最坏情况发现标准测试难以暴露的深层问题
- **模型漂移和数据漂移是线上服务的常态**：需要建立完善的"采集→监控→分析→迭代"闭环
- **A/B 测试是模型上线前的最后把关**：小流量验证新版本后再全量发布
